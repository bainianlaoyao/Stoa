# 愿景与设计原则

## 项目愿景

Vibecoding 极简多会话管理框架是一个面向 AI 并发编程场景的本地调度台。它的职责不是替代 IDE，也不是实现一个通用终端复刻品，而是为“多终端、多 Agent、多工作区、低状态丢失风险”的工作方式提供一个更稳定、更克制的桌面容器。

这个框架最重要的价值不在于视觉花哨，而在于把三个高频痛点变成系统能力：

1. 会话不轻易丢失。
2. UI 切换不打断终端工作流。
3. Agent 内部状态能通过可靠侧信道反馈给 GUI，而不是靠脏解析猜测。

## 核心设计原则

### 1. 拒绝脏解析

终端字符流只负责给人看，不负责给机器做状态推断。所有 Agent 生命周期、工具调用、错误信号、真实会话指针都必须通过结构化侧信道回传，优先使用 CLI 官方 hook 或外挂脚本的 JSON 事件。

### 2. 前端极度降智

Vue 渲染层只承担状态映射与指令转发，不拥有会话的真实控制权。工作区状态、PTY 生命周期、Webhook 接收、状态合成、恢复逻辑全部驻留在 Node.js 主进程或其受控子模块内。

### 3. 原生机制代理

终端展示交给 `node-pty` 与 `xterm.js`；会话上下文恢复交给 CLI 自身的 session 能力；桌面壳交给 Electron。系统的价值在于调度与边界管理，而不是重复实现底层机制。

### 4. 直觉优于极客

用户交互优先选择低认知负担的点击式工作区切换，而不是高度依赖快捷键记忆或全局命令面板。设计上要偏向“上手立即理解”，而不是“熟练后极限高效”。

### 5. 白盒优于沙箱

扩展机制不采用复杂插件沙箱，而采用明确目录分区下的白盒扩展。扩展模块可直接复用主工程状态与组件能力，以换取实现速度、调试透明度和低集成成本。

## 本阶段工程约束

- 先定义文档与边界，再启动工程实现。
- 先保证架构路径正确，再追求一次性做全。
- 任何实现都不得破坏双通道模型和 dumb UI / smart backend 原则。
