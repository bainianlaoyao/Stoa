Current Failure

Why the present model is structurally fragile

The current system stores runtime facts in workspace hook artifacts. That makes port rebinding, session ownership, and secret validity impossible to keep coherent once STOA restarts or another instance touches the same workspace.

RC-1

Claude HTTP sidecars hardcode a concrete localhost port. Restarting STOA turns the hook URL stale immediately.

RC-2

Manual terminal launches do not carry STOA runtime env. Hooks either fail loudly or disappear silently.

RC-3

One workspace file can express only one owner route. The last sidecar rewrite wins, even if another STOA still owns live sessions.

RC-4

Webhook binding uses an OS-assigned port. Every restart changes the address, but running agents keep using old assumptions.

Workspace Files

Static sidecars carry runtime truth

.claude/settings.json and related artifacts embed the current webhook route even though it belongs to a live session owner.

Launch-Time Binding

Route chosen too early

The destination port and secret assumptions are fixed when sidecars are written or when an agent process first starts.

Webhook Server

Receiver moves, files do not

STOA can restart cleanly and bind a fresh port, but the workspace files and already-running agents do not share the same view anymore.

Target Topology

Final lease-driven dispatch architecture

Workspace hook artifacts stop owning routing state. They only invoke a stable dispatcher. The dispatcher reads the current session lease, discovers the active STOA owner, and posts to the correct webhook receiver in real time.

Workspace Hook Surface

Long-lived, provider-facing trigger files. Stable across STOA restart.

Lifecycle

Session hook lease lifecycle

The session lease is the only moving routing artifact. That is why STOA restart becomes survivable without rewriting workspace hook files or relying on agents to hot-reload configuration.

01

Acquire

STOA claims the session lease before spawning the provider runtime.

ownerInstanceId generation webhookBaseUrl sessionSecret
02

Spawn

The provider receives only a stable lease pointer, not live routing state.

STOA_HOOK_LEASE_PATH
03

Dispatch

Every hook invocation reads the current lease and discovers the live STOA owner at that moment.

late-bound route resolution
04

Heartbeat

The owner refreshes heartbeatAt and expiresAt on a short cadence.

suggested: 5s / 20s
05

Reclaim

After expiry, a new STOA owner reclaims the same session with a new secret and incremented generation.

generation + 1 secret rotates
06

Release

On clean stop, archive, delete, or session exit, the lease is deleted or forced stale.

dead lease = dead route
Ownership Boundary

How multi-STOA stays safe on one workspace

Shared workspace artifacts are intentionally stable and boring. The real isolation boundary is session ownership. Multiple STOA instances may coexist on the same workspace, but one live session still gets only one fresh owner.

Shared workspace stays static

The workspace can be touched by multiple STOA instances because the hook surface no longer encodes live port or owner information.

.claude/settings.json -> command hook only .codex/hooks.json -> command hook only .opencode plugin -> dispatcher contract only .stoa/hook-dispatch.mjs -> stable bridge artifact

Session ownership stays exclusive

Each session lease names the current owner instance. A fresh lease blocks another STOA from taking over the same session until the lease expires or is released.

Invariants

The rules this architecture refuses to break

These are the properties that make the system robust instead of merely patched. If any one of them is violated in implementation, the design regresses toward the current stale-port model.

No runtime truth in workspace

Sidecar artifacts may express events and commands, but never port, secret, lease freshness, or owner identity.

Dispatch is always late-bound

Route resolution happens at hook fire time, not sidecar install time and not process boot time.

One fresh owner per session

Session control is exclusive. Workspace access can be shared; live session ownership cannot.

Secret rotates on reclaim

New ownership means new authorization. Old secrets must stop being valid immediately after reclaim.

Manual CLI stays quiet

Unmanaged terminal launches should degrade to safe no-op rather than visually broken hook noise.

Restart self-heals through lease refresh

Running agents keep the same dispatcher path. Future hook invocations read the new lease and find the new STOA owner automatically.

Visual language follows the repository's Modern Minimalist Glassmorphism + Clean UI direction and uses the shared token vocabulary from the renderer token system.
