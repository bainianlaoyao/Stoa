# 系统总体架构

## 总览

系统采用 Electron 作为桌面外壳，主进程负责真实状态与进程协调，Vue 渲染进程只做界面映射。终端输出与状态事件被拆成两条独立通道，分别服务于“人类观察”和“机器判断”。

## 运行责任域

### 1. Electron Main Process

主进程负责：

- 创建与管理 `BrowserWindow`。
- 托管 PTY 与 CLI 子进程。
- 承载 Express webhook server。
- 维护 Session Manager。
- 管理状态文件的读写与恢复。

### 2. Preload Bridge

Preload 层只暴露白名单 API 给渲染层，例如：

- 获取工作区列表。
- 发送终端输入。
- 订阅工作区状态事件。
- 触发工作区切换、重启和尺寸调整。

### 3. Renderer Process

渲染层基于 Vue 3 + Pinia + xterm.js 实现。它不拥有真实会话控制权，只持有后端镜像状态，并将用户意图上送主进程。

### 4. Hook Sidecar / CLI Runtime

外挂脚本在 CLI 运行态中拦截结构化事件，并通过 webhook 回传主进程。CLI 自身负责内部会话历史和上下文恢复，不由 GUI 复制实现。

## 核心原则如何落到架构上

- “拒绝脏解析”对应双通道分离。
- “前端极度降智”对应主进程集中状态控制。
- “原生机制代理”对应对 Electron、PTY 和 CLI 自有能力的复用。
- “白盒优于沙箱”对应内部扩展目录与共享状态访问。

## 后续实现边界

未来代码实现必须确保：

- `node-pty` 只存在于主进程或受控后端模块。
- CLI 会话指针只由主进程存储和传递。
- Renderer 不直接访问文件系统或子进程能力。
