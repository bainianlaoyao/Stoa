# 白盒扩展模型

## 设计动机

系统不采用重量级插件沙箱机制，因为当前目标不是构建一个第三方应用市场，而是让内部演进足够快。白盒扩展通过明确目录与边界约束，换取更直接的开发体验和更低的集成成本。

## 扩展类型

### 1. Provider 扩展

位于 `src/extensions/providers/`，负责：

- 不同 CLI 的命令行适配。
- hook sidecar 注入策略。
- session id 抓取与恢复参数拼装。

### 2. Panel 扩展

位于 `src/extensions/panels/`，负责：

- 附加信息展示。
- 调试辅助面板。
- 与工作区关联的次级视图。

## 可访问能力

白盒扩展可在受控前提下访问：

- 共享 Pinia store 的投影状态，以及受约束的本地视图状态。
- 约定的主进程 API。
- 公共类型与事件模型。

## 不可做的事情

- 不得重写核心恢复逻辑。
- 不得私自定义与主状态冲突的工作区生命周期。
- 不得把实验能力直接注入核心目录而不经过扩展边界。

## 权衡

白盒扩展提升开发效率，但安全隔离更弱，因此只适用于可信代码环境。这个项目明确接受这一权衡，因为目标是内部高频演进，而不是开放生态。即便如此，扩展也不得把 renderer store 重新升级成真实状态来源。
