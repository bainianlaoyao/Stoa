## Context Report: Provider Built-in Title Strategy

### Why This Was Gathered
Assess whether Stoa already has a viable way to obtain built-in session titles from all three providers through hooks, and if not, evaluate whether title generation should move to a Stoa-owned provider path via `stoa-ctl` and user-supplied API credentials.

### Summary
Stoa does not currently have a complete scheme for obtaining built-in titles from all three providers through hooks. The current authoritative work-session title is still generated by Stoa at session creation time, and there is no host-side state patch path that can update `session.title` from provider events after the session starts.

Provider capabilities are asymmetric. Claude Code can set a session title from a `UserPromptSubmit` hook response, OpenCode has its own built-in title generator and persists the result in its local SQLite `sessions.title`, while the current Codex integration exposes custom model-provider configuration and hook lifecycle events but no stable built-in session-title field in the hooked data path Stoa is using today. Given that asymmetry, the recommended product direction is to keep title generation host-owned in Stoa rather than make session naming depend on provider-native title behavior.

### Key Findings
- Stoa currently treats work-session title generation as host-owned logic at creation time.
- Stoa has no existing event-to-title sync pipeline; hook events can update state, summary, model, snippet, and external session id, but not `session.title`.
- Claude Code supports setting a session title from hook output, but Stoa does not currently consume or project that capability.
- OpenCode has first-class built-in title generation and stores the resulting title in SQLite, but Stoa does not currently read it.
- Codex’s current Stoa integration does not expose a comparable built-in title source through the existing hook adapter path.
- Codex officially supports custom model providers with user-supplied API keys via config, so a Stoa-owned title-generation provider is technically feasible if you want a dedicated naming backend.

### Risks / Unknowns
- [!] Even if Claude or OpenCode can produce provider-native titles, Stoa cannot persist them without adding a new host-owned title sync path.
- [!] Using provider-native titles as the primary naming contract would create asymmetric behavior across Claude, OpenCode, and Codex.
- [!] A dedicated Stoa-owned title provider would add another external API dependency and auth surface, even if it is technically straightforward.
- [?] Codex may have internal thread naming outside the current Stoa path, but no stable title field was identified in the hooked data Stoa consumes today.
